## 太长不读

这篇写给产品、业务、运营的同学，不讲老套的 Prompt 技巧，想和大伙讲怎么把东西真正用 AI 做出来：`CLAUDE.md` 怎么写、需求怎么描述清楚、Plan 模式和 Skills 怎么用、非技术人做开发要注意哪些坑这些问题。

大多数人用不好 Claude Code 的原因和技术能力关系不大，真正决定结果的是上下文有没有交代清楚、工作规范有没有说明白、经验有没有沉淀下来。

---

## Claude Code 实际在做什么

给 Claude Code 一个任务，它不是生成一段代码让你粘贴，而是自己打开项目文件读，分析依赖关系，写代码，跑一遍看有没有报错，有问题自己修，循环几轮，停下来告诉你结果。整个过程你不需要手动搬运任何内容。

和在对话框里让 AI 写代码的区别不在于模型聪明程度，是执行方式根本不同。对话框是来回搬运：你描述、它生成、你判断、你粘贴、你跑、有问题再描述。Claude Code 是直接执行：它在你的项目目录里运行，自己读文件、改文件、跑命令、查报错，跑完整个循环出结果了才来找你确认。

<img src="https://gw.alipayobjects.com/zos/k/y3/chatgpt_vs_claude_code_workflow.svg" width="1000px" alt="对话框 AI 和 Claude Code 的执行方式对比：前者需要来回搬运，后者直接在项目中执行完整循环">

<img src="https://cdn.fliggy.com/upic/QrTwEC.svg" width="1000px" alt="Agent Loop">

Claude Code 跑的是一个反复循环的代理过程：

```
收集上下文 → 采取行动 → 验证结果 → [完成 or 回到收集]
     ↑                    ↓
  CLAUDE.md          权限 / 沙箱
  Skills             Tools / MCP
  Memory
```

终端用系统自带的就够，想要体验好一点可以装 [Ghostty](https://ghostty.org)。安装一条命令：

```bash
npm install -g @anthropic-ai/claude-code
```

进项目目录跑 `claude` 就可以开始了。复杂任务它还会自己拆出子代理并行处理，终端里能看到类似 `[subagent] analyzing dependencies...` 的输出，说明它在同时推进多件事。

这个执行循环很强，但有一个前提：Claude Code 每次启动都对你的项目一无所知，不告诉它任何背景，它就只能猜你用什么框架、猜命名规范、猜怎么构建。猜对了是运气，猜错了你就要返工。

## CLAUDE.md：每次启动都会读的那个文件

很多人装好 Claude Code 之后直接开始问问题，结果每次都要重新交代背景，用一会就觉得很烦。原因几乎都一样：没有建 `CLAUDE.md`。

<img src="https://cdn.fliggy.com/upic/1auitO.svg" width="1000px" alt="Claude Code 的上下文加载层级：CLAUDE.md 是每次都常驻的最底层，Skills 和工具按需加载">

`CLAUDE.md` 放在项目根目录，Claude Code 每次启动都会先读它，把里面的内容作为当前项目的背景和规则加载进来。类比一下，这是你给新同事写的项目交接文档，区别是 Claude Code 每次都会从头认真读，而且严格执行。

没有 `CLAUDE.md` 时，让 Claude Code 帮你给用户表加一个 `status` 字段，它会猜你用什么数据库、猜字段类型、猜迁移方式，可能给你一个和项目完全对不上的方案。有了 `CLAUDE.md` 之后，它知道你用的是 PostgreSQL、ORM 是 Prisma、迁移文件要放哪、命名规范是什么，直接给你可以跑的东西。

对非技术人来说，`CLAUDE.md` 里值得写的内容大概是这些：

```markdown
# 项目背景
这是一个面向中小企业的 CRM 系统，技术栈是 Python + Django，前端用 Vue3。
产品经理是 Alice，设计师是 Bob，后端是 Carol。

# 工作规范
- 所有代码注释用中文
- 新功能先写实现，不主动加测试，除非我明确要求
- 修改文件前先说明你打算改什么，确认后再动

# 命名规范
- 变量和函数用英文，类名用 PascalCase
- 数据库表名用下划线分隔，如 user_profile

# 禁止项
- 不要主动重构我没有让你改的代码
- 不要删除任何文件，除非我明确说"删掉它"
```

Claude Code 的上下文总量是 200K tokens，CLAUDE.md 通常占其中 2,000-5,000 tokens。Anthropic 官方自己的 CLAUDE.md 也只有约 2.5K tokens。保持在 150 行以内，背景、规范、禁止项写清楚，细节放到具体 Skill 里。把所有文档和历史决策全堆进去，会持续挤压后续对话的可用空间。

CLAUDE.md 解决了项目级别的背景，但每个具体任务还需要单独说清楚。同一个需求，描述精不精确，结果差的可不止一点。

## 一个案例讲清楚：需求描述的精确度

你描述需求的精确度，直接决定了 Claude Code 能帮你做到什么程度。写 PRD 也是同一个逻辑，模糊的需求给到工程师，做出来的东西往往不是你要的。来看一个真实对比。

**模糊版本：**

```
帮我做一个语音工具
```

给这个需求，Claude Code 可以做出来点东西，但它会猜很多：是网页版还是桌面应用？什么系统？怎么触发录音？录完之后做什么？支持什么语言？最后给你的很可能是一个简单的网页麦克风演示，和你真正想要的差十万八千里。

<img src="https://gw.alipayobjects.com/zos/k/vm/vague_vs_precise_prompt_paths.svg" width="1000px" alt="模糊需求让 Claude Code 分叉猜测，最终偏离目标；精确需求直接命中，产出可用">

**精确版本：**

```
帮我做一个 macOS 原生语音输入工具，用 Swift 开发，具体要求如下：

1. 按住 Fn 键开始录音，松开后将转录文字注入当前聚焦的输入框。
   优先使用流式转录（Apple Speech Recognition framework）。
   Fn 键通过 CGEvent tap 全局监听，需抑制 Fn 事件传递以防止触发 emoji 选择器。

2. 默认语言必须为简体中文（zh-CN），确保开箱即用就能识别中文输入。
   同时在菜单栏提供语言切换选项（英语、简体中文、繁体中文、日语、韩语）。
   语言选择存储在 UserDefaults 中。

3. 录音时在屏幕底部居中显示一个无边框胶囊状悬浮窗，不要有红绿灯和 titlebar。
   使用 NSPanel（nonactivatingPanel）+ NSVisualEffectView（.hudWindow 材质）。
   高度 56px，圆角半径 28px，包含：
   - 左侧 5 根竖条波形动画（44×32px），由实时音频 RMS 电平驱动，
     竖条权重 [0.5, 0.8, 1.0, 0.75, 0.55]，attack 40%，release 15%，
     每根竖条 ±4% 随机抖动
   - 右侧文字标签（160-560px 弹性宽度）实时显示转录文本
   - 入场弹簧动画 0.35s，文字宽度过渡 0.25s，退场缩放动画 0.22s

4. 文字注入用剪贴板 + 模拟 Cmd+V。注入前检测当前输入法：
   如果是 CJK 输入法，先临时切换到 ABC 键盘再粘贴，完成后恢复原输入法。
   注入完成后恢复原剪贴板内容。

5. 接入 LLM 提升识别准确率，处理中英文混杂场景。
   通过 OpenAI 兼容 API（可配置 Base URL、Key、Model）对转录文本做 refine。
   system prompt 要求极保守：只修复明显的语音识别错误（如"配森"→"Python"、
   "杰森"→"JSON"），绝不改写或润色看起来正确的内容，正确就原样返回。

6. 菜单栏提供 LLM Refinement 子菜单，含启用/禁用开关和 Settings 入口。
   Settings 窗口有 API Base URL、API Key、Model 输入框，含 Test 和 Save 按钮。
   松开 Fn 键后如果 LLM 已启用，悬浮窗显示 Refining... 状态，等返回后再注入。

7. 应用以 LSUIElement 模式运行（仅菜单栏图标，无 Dock 图标）。
   使用 Swift Package Manager 构建，提供 Makefile（build/run/install/clean），
   构建产物为签名的 .app bundle。
```

两个需求描述给 Claude Code，结果差的不是质量好一点差一点，是能不能直接用。精确版直接产出一个可以装的 macOS 应用，细节都对。

比写详细点这个结论更值得看的，是每一条细节背后防止了什么猜测：

| 需求中的细节                  | 如果不写，Claude Code 会怎么猜          |
| ----------------------------- | --------------------------------------- |
| macOS 原生 + Swift            | 可能做成 Python 网页版或 Electron 应用  |
| Fn 键触发、全局监听           | 可能做成点按钮录音，或只在应用内有效    |
| 抑制 Fn 传递到系统            | emoji 选择器会被触发，体验破坏          |
| 默认简体中文                  | 默认英文，用中文说话识别率极差          |
| CJK 输入法切换再粘贴          | Cmd+V 被中文输入法拦截，文字注入失败    |
| LLM 纠错"极保守"的明确要求   | 可能过度润色，改掉你原本想说的意思      |

上面这份语音工具的需求描述，本质上已经是一份结构化的 PRD：技术栈是 Swift + macOS，触发是 Fn 键 + CGEvent tap，UI 细到 56px 高、28px 圆角、波形权重数组，异常处理到 CJK 输入法切换和剪贴板恢复。Claude Code 拿到之后几乎不需要猜，直接出结果。

但业务级的 PRD 比工具需求多三层，换成我要写飞猪国际门票频道页这类需求，就必须补上的：

**一、先写要解决的问题，不是功能描述**

语音工具的需求之所以精确，是因为场景非常清晰：在任意输入框里按住 Fn 键录音，松开后把文字填进去。但业务 PRD 里这个背景往往不说，Claude Code 就不知道为什么要做这个东西，碰到歧义只能猜。比如说我要写国际门票频道页，要先写：

```
国际门票目前没有独立入口，用户只能通过搜索触达，非热门城市曝光率很低，
转化漏斗在"发现"环节就断了。目标是建一个频道页让用户按城市浏览，
提升非搜索场景的转化率。
```

这两句话决定了 Claude Code 在碰到"热门城市展示几个"、"筛选项要不要做'最近浏览'"这类问题时的判断方向。

**二、范围用"做/不做"两栏写死**

语音工具里隐含了不做的东西，比如没提录音文件保存、没提历史记录、没提云端同步，Claude Code 就不会加。但业务 PRD 边界更模糊，Claude Code 很积极，你说做一个列表页，它可能顺手给你加上收藏按钮、分享入口、埋点上报。不做的部分必须明确列出来：

```
不做：商品详情页（已有，直接跳转 /detail/{itemId}）
不做：登录态功能（收藏、历史、个性化推荐）
不做：分享、SEO、埋点（下一期）
```

**三、异常和边界情况单独一节**

语音工具里每一条都写了异常处理，比如第 4 条的 CJK 输入法切换、第 5 条的 LLM 纠错保守策略。业务 PRD 里这块通常被省掉，Claude Code 正常流程能做好，区别就在这里。接口超时、数据为空、图片挂掉、搜索无结果，每种情况写清楚处理方式，不然 Claude Code 要么不处理，要么猜一个你不满意的兜底方案。

**四、验收标准要有可测量的数字**

语音工具隐含了验收标准，比如按下 Fn 键有波形动画、松开后文字出现在光标处、CJK 环境下注入不失败。业务 PRD 要显式写出来，而且要给数字：

```
首屏 LCP < 1.5s（Wi-Fi 环境）
页面在 375px 和 428px 宽度下布局正常
滚动到底部距离 200px 时触发下一页加载
接口超时 >3s 展示骨架屏 + 重试按钮
```

没有数字的验收标准，Claude Code 只能自己判断"差不多就行"。

**五、PRD 里不能出现"待定"**

TBD、TODO、后续再看，Claude Code 碰到这些会自己猜着填，猜的往往不是你要的。每一处决策点都写清楚，哪怕写的是"这个版本先硬编码，下版本再做配置化"，也比空着强得多。

这五条可以做成一个 Skill 沉淀下来，把 PRD 模板写进去，之后每次写需求让 Claude Code 自动按这套结构来，不用反复交代格式。

## Plan 模式：先对齐再执行

Claude Code 默认会直接动手，你说一句话它就开始改文件。大多数时候这挺好，效率高。但碰到复杂任务，这个习惯会出问题。

有次让它直接"把登录模块重构一下，统一一下错误处理"，没有用 Plan 模式，它改了 8 个文件，删了一个当时没用但后来会用的 util 文件，还顺手修改了 3 处和登录完全无关的地方。回滚花了半小时。

Plan 模式（计划模式）的作用就是在这之前加一道确认：按 `Shift+Tab` 两次切换进去，Claude Code 会先列出它打算怎么做，你确认方向对了再执行。类比工作场景，不是直接让小李去把功能做掉，而是先拉个会，看看方案合不合理，觉得 ok 了再动手。

<img src="https://gw.alipayobjects.com/zos/k/tb/normal_vs_plan_mode_flow.svg" width="1000px" alt="普通模式直接执行可能偏差，Plan 模式加入确认环节，结果更可控">

Plan 模式输出的计划长这样：它列出要改哪几个文件，每个文件改什么，改的理由是什么，预计会影响哪些地方。用业务逻辑来判断这个方向对不对，比判断代码本身容易得多。

Plan 模式解决单次任务的对齐。但有些东西你会反复用到，PRD 模板、上线检查清单、周会整理格式，每次重新交代一遍太低效，这就是 Skills 要解决的。

## Skills：只有调用时才完整加载

Skills（技能包）是 Claude Code 里最被低估的功能之一。

先说机制：Claude Code 启动时只读每个 Skill 的 frontmatter（通常 100-200 tokens），真正调用某个 Skill 时才加载完整内容。这意味着你可以装几十个 Skill，启动开销几乎不增加，不会挤压对话上下文。

<img src="https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69cd895f7f04456cccf7b7e0_image3.png" width="1000px" alt="Skills 渐进式加载：Claude Code 启动时只读描述，实际使用时才加载完整内容">

Skill 是文件夹，不是单个文件。`SKILL.md` 定义任务边界和执行步骤，其他素材（参考文档、模板、脚本）放到同级目录里，需要时按需读取，不会一次性全塞进上下文：

```text
.claude/skills/
└── 整理周会/
    ├── SKILL.md           ← 触发条件 + 执行框架
    ├── output-template.md ← 输出模板，内容长就单独放
    └── examples/
        └── sample.md      ← 示例，仅需要时读取
```

### Skill 的三种类型

**检查清单型**：每次做某件事之前跑一遍，确保不漏项。比如需求上线前的自查：

```markdown
---
name: 需求上线检查
description: 需求发布前跑一遍，确认没有遗漏
---

## 上线前必须全部通过
- [ ] PRD 里的验收标准逐条确认
- [ ] 设计稿和实现对齐，间距、文案、交互没有遗漏
- [ ] 异常状态（空态、报错、超时）都有对应处理
- [ ] 数据埋点按规范打好
- [ ] 测试环境验证通过

## 输出
每项 Pass / Fail，有 Fail 的必须修完再发布。
```

**工作流型**：固定步骤的操作，写一次反复跑。比如整理周会记录：

```markdown
---
name: 整理周会
description: 开完会有原始记录需要整理时调用
---

## 输入
原始会议记录（录音转录、手写笔记、会议软件导出均可）

## 输出格式
**本周达成**：[负责人] 完成了什么，只写明确完成的事
**下周计划**：[负责人] 做什么，截止时间
**待讨论**：卡在哪里，需要谁来决定
**行动项**：[谁] [做什么] [什么时候]，只列有明确后续动作的

## 规则
- 不润色，保持原始措辞
- 记录不完整标注"信息缺失：XX 不清楚"
- 不确定的标注"待确认"，不要猜
```

**领域专家型**：把某个领域的判断框架封装进去，Claude Code 碰到这类问题时按固定路径走，不每次自由发挥。比如线上问题排查：

```markdown
---
name: 线上问题排查
description: 收到线上告警或用户反馈有异常时调用
---

## 收集信息
1. 报错截图或错误日志的完整内容
2. 影响范围：哪些用户、哪个页面、什么时间开始
3. 最近的变更：代码发布、配置修改、数据变更

## 判断矩阵
| 现象       | 优先检查                     |
| ---------- | ---------------------------- |
| 页面白屏   | JS 报错 → 最近发布记录       |
| 接口超时   | 服务监控 → 数据库慢查询      |
| 数据异常   | 最近数据变更 → 上下游依赖    |

## 输出格式
根因 / 影响范围 / 修复步骤 / 验证方式
```

### description 写触发条件，不写功能说明

description 常驻上下文，越短越好，而且要写"何时该用我"，不是"我是干什么的"：

```yaml
# 不好（说的是功能，约 15 tokens）
description: 把会议录音转录或原始记录整理成结构化周报

# 好（说的是触发条件，约 9 tokens）
description: 开完会有原始记录需要整理时调用
```

### 调用频率决定要不要自动触发

Skills 有自动和手动两种触发方式：

- 每次会话都会用的 → 保持自动，description 写精准
- 偶尔才用的 → 设置 `disable-auto-invoke: true`，手动叫它，description 完全不占上下文
- 几乎不用的 → 直接移除，改写成 `CLAUDE.md` 里的一条说明

### 常见写法问题

- description 太模糊：`帮你处理后台相关的事`，什么都能触发
- 正文太长：几百行手册全塞进 `SKILL.md` 正文，应该拆到 supporting files
- 一个 Skill 做太多事：审查、发布、调试、文档全写在一起，拆开才用得准
- 有副作用的 Skill 允许自动调用：会修改文件或执行命令的 Skill，必须加 `disable-model-invocation: true`，不然 Claude Code 会自己判断要不要跑

description 里补上"什么时候不该用这个 Skill"的反例，准确率提升非常明显，有测试数据显示没有反例时准确率是 53%，加上之后升到 85%，响应速度还快了 18%。

社区里有很多现成的 Skill 可以直接装，不用自己从头写。比如 [ljg-skills](https://github.com/lijigang/ljg-skills) 里有几个挺有意思的：

`ljg-roundtable`：说一句"圆桌讨论 AI 到底有没有创造力"，Claude Code 会自动召集相关领域的代表性人物，分配立场，组织一场有来有回的结构化辩论，每个角色都有自己的逻辑。想快速从多个角度看透一个问题，比自己一个人想要快。

`ljg-travel`：给一个城市名，它帮你做出发前的深度文化功课——博物馆重点、历史背景、古建筑脉络，参考考古学 Desk-Based Assessment 的方法，穷尽文献再去现场。最后输出一份结构化研究文档和可以带走的参考卡片。

到这里，你已经知道怎么配置项目背景、怎么描述需求、怎么对齐方案、怎么沉淀经验。但用久了你会碰到一个问题：同一个会话跑时间长了，Claude Code 的表现会逐渐变差。

## 上下文会用完：长会话为什么越跑越偏

Claude Code 的工作台大小是固定的，大约 200K tokens。你们之间所有的对话、它读过的文件、跑过命令的输出，都在这张桌子上。跑的时间越长，早期的内容就越往边上挤，最后被推下去。

被推下去的不只是不重要的东西。压缩时系统优先保留可以重新读取的内容，比如文件。但有一类东西很难恢复：决策的理由。你之前告诉它"这个接口要用 POST 不用 GET，因为数据量大"，压缩之后可能只剩"用 POST"，理由没了。两小时后你问相关问题，它可能给你一个完全不同的方案，好像之前的讨论从没发生过。

<img src="https://cdn.fliggy.com/upic/HQlqLw.svg" width="1000px" alt="会话越长，早期决策越容易被压缩丢失，产生前后矛盾">

有几个应对方式：

**CLAUDE.md 里写 Compact Instructions**。告诉 Claude Code 压缩时什么要留：

```markdown
## Compact Instructions
压缩时按优先级保留：
1. 架构决策和理由（不要摘要，保原文）
2. 改了哪些文件，改了什么
3. 当前进展状态
4. 未完成的 TODO
```

**长会话结束前让 Claude Code 写交接笔记**。直接说："把当前进度写成一份 HANDOFF.md，包括做了什么、试过什么没成功、下一步该做什么。"下次开新会话时把这个文件路径发给它，就能接着做，不依赖压缩算法保留了多少。

**任务切了就 `/clear`**。这不只是个习惯，是因为新任务和旧任务之间的上下文经常互相干扰，清掉是最干净的。

理解了这些，来看实际上这套东西用在哪些场景最顺手。

## 非技术人真正适合用 Claude Code 的场景

**做一个小原型或内部工具**

有个同学想做一个简单的页面来可视化团队的 OKR 完成情况，原来要排期等工程师。自己把需求描述清楚：数据从哪来、展示什么指标、界面大概什么样，Claude Code 直接做出一个可以用的版本，一天之内。不是生产级别的，但验证想法完全够用，等真正要上线再交给工程师打磨。

关键是把场景描述清楚：这个工具给谁用、要做什么、什么格式的数据进来、什么格式的结果出去。

**处理和分析数据**

你有一份 Excel 或 CSV，想按月份汇总销售额，或者找出哪个渠道的转化率最低，或者把多个表格合并成一个。这类任务 Claude Code 做得很快，你只需要说清楚数据的结构和你想要的结果格式。

比我想分析一下数据有用得多的描述是：告诉它 CSV 有哪几列、按什么维度分组、算什么指标、最后输出什么格式。

**竞品调研和外部数据获取**

让 Claude Code 去爬取几个竞品的公开信息、对比功能列表、整理成表格，处理起来效率很高。不能做需要登录或付费的内容，但公开的页面、文档、发布说明都没问题。

这里可以搭配 [opencli](https://github.com/jackwener/opencli)，它把各类网站封装成 Claude Code 可以直接调用的命令行接口（CLI）。比如 [xiaohongshu-cli](https://github.com/jackwener/xiaohongshu-cli) 装好之后：

```bash
pipx install xiaohongshu-cli              # 安装
npx skills add jackwener/xiaohongshu-cli  # 给 Claude Code 安装对应 Skill
xhs login                                 # 从浏览器提取 Cookie，登录一次
xhs search "露营装备" --sort popular       # 搜笔记，结果直接可以让 Claude Code 分析
xhs hot -c travel                         # 按分类抓热门
```

<img src="https://cdn.fliggy.com/pic/9NoUBj09.png" width="1000px" alt="用 xiaohongshu-cli 搜索小红书，结果以结构化格式输出，直接交给 Claude Code 分析">

拿到结构化数据之后，再让 Claude Code 做分类、提炼热词、和自家产品做对比都非常顺畅。GitHub 的 `gh` CLI 也是同样的思路，装好之后 Claude Code 能直接查 issue、看 PR，不需要打开浏览器。

更多使用场景可以看 Anthropic 官方整理的 [Use Cases](https://claude.com/resources/use-cases)，涵盖 Marketing、Finance、HR、Legal、Research 等类别，找找有没有和自己工作直接对得上的。

## 如果想进一步：如何构建自己的 Agent

用 Claude Code 用久了，你可能会想：能不能给特定业务场景搭一个专用的自动化助手？不是通用的 Claude Code，而是一个只做这件事、会自动触发、不需要人在旁边守着的东西？这就是 Agent 的用武之地。

Agent 和 Claude Code 背后是同一套机制，区别在于 Claude Code 是个通用工具，而你搭的 Agent 可以针对具体场景裁剪，只配它需要的工具，只让它处理规定范围内的任务。

### 先搞清楚：Workflow 还是 Agent？

很多标着"Agent"的产品，拆开看其实是 Workflow，弄清楚这两者的区别，有助于你在和工程师沟通时给出正确的需求方向。

| 维度       | Workflow                           | Agent                              |
| ---------- | ---------------------------------- | ---------------------------------- |
| 控制权     | 代码预定义，同输入必走同一路径     | LLM 动态决策，步骤可以自己调整     |
| 执行方式   | 工具顺序固定，错误走预设分支       | 工具按需选择，可以自我修复         |
| 维护成本   | 改流程需修改代码并重新部署         | 调整说明文字即可，无需重新部署     |
| 适用场景   | 流程固定、输入边界清晰             | 需要中间推理与灵活判断             |

<img src="https://gw.alipayobjects.com/zos/k/dh/workflow_vs_agent.svg" alt="Workflow 与 Agent 对比" width="1000" />

日常能自动化的大多数业务场景，其实 Workflow 就够了：每周定时生成报表、收到邮件自动分类回复、表单提交后触发审批链。流程是固定的，边界是清晰的，这类场景不需要模型动态推理，直接用工具链串起来更稳定。

只有当任务本身需要中间判断，比如根据用户反馈类型决定下一步处理方式、复杂客服问题需要查多个数据源再综合答复，才真正需要 Agent。

### 五种控制模式

大多数 AI 自动化系统拆开看，都是这五种模式的组合，很多场景不需要完整的 Agent，几种模式搭在一起就够：

1. **提示链 Prompt Chaining**：任务拆成顺序步骤，每步处理上一步的输出。适合线性流程，比如先生成大纲、再写正文、再翻译。

2. **路由 Routing**：对输入分类，定向到对应的处理流程。简单问题走轻量模型，复杂问题走强模型；技术咨询和账单查询走不同逻辑。

3. **并行 Parallelization**：把独立子任务同时跑，或者同一任务跑多次取共识。适合高风险决策或需要多视角的场景。

4. **编排器-工作者 Orchestrator-Workers**：主 LLM 动态分解任务，分发给多个工作者各自处理，最后综合结果。适合任务边界不固定、需要动态分工的场景。

5. **评估器-优化器 Evaluator-Optimizer**：生成器产出，评估器给反馈，循环直到达标。适合翻译、内容审核这类质量标准难以用代码精确写死的任务。

<img src="https://gw.alipayobjects.com/zos/k/dw/five_agent_patterns.svg" alt="五种常见控制模式" width="1000" />

对产品和业务同学来说，认识这五种模式的价值不是自己去实现，而是在给工程师提需求时能说清楚你要的是哪种结构，而不是只说"做一个 AI 助手"。

## 几条用下来有用的技巧

**一个任务一个会话，做完就 `/clear`**。前面讲了上下文会用完，这是最直接的应对方式。做完一件完整的事就清，新任务开新会话，不让两件事的上下文互相污染。一个会话里同时推进三个任务，不如三个会话各推进一个。

**Memory：让 Claude Code 记住你的偏好**。`CLAUDE.md` 是项目级别的规范，每个项目都有自己的；Memory 是用户级别的，跨所有项目跨所有会话都生效。你可以告诉 Claude Code "记住我喜欢先看方案再执行"、"记住我用的是 macOS"、"记住回复用中文"，它会把这些偏好写进 `~/.claude/memory/`，下次打开任何项目都能记住。经常交代的背景信息都可以沉淀进去，不用反复说。

**截图能解决很多描述困难**。要描述一个界面问题或者想参考某个设计风格，直接截图丢给 Claude Code，比用文字描述省力，效果也更准。

**把完成标准说出来**。做完了对 Claude Code 来说是模糊的，把页面加载不报错、数据能正常显示、点保存有成功提示这类标准说出来，它才知道自己什么时候可以停。

**手机派任务，Claude Code 在 Mac 上帮你干完**。之前 Claude Code 的远程控制是你从手机接管终端会话，能做的事局限在终端里。新出的 Dispatch + Computer Use 不一样：Claude Code 可以直接操控你 Mac 的图形界面，打开任何应用、在浏览器里点来点去、往表格里填数据，不只是跑命令。需要打开某个网页填个表、在 Excel 里改几列数字这类活，它现在真的能做了。还支持定时任务，跟它说一次每周五下午把本周数据整理出来，之后就不用再管。

<video src="https://cdn.fliggy.com/pic/claude-iPhone-control-mac44.mp4" width="1000px" controls></video>

**Hooks 是更进一步的自动化**。Hooks（钩子）在特定事件发生时自动执行某些动作，比如每次改完代码自动运行格式检查，或者每次会话开始自动加载某些上下文。稍微需要一点设置，配好之后能省掉很多手动步骤。

<img src="https://cdn.fliggy.com/upic/5J83L4.png" width="1000px" alt="Hooks 配置界面，可以在会话开始、工具调用前后、任务完成等节点插入自定义逻辑">

**`/simplify`：改完代码立刻跑一遍**。它会对刚改的代码做三个维度的检查：代码复用、质量、效率，发现问题直接修掉。不用手动 review，改完就执行，很省事。

**双击 ESC 重新编辑上一条**。Claude Code 跑偏了，或者上一句没说清楚，按两次 ESC 可以回到上一条输入重新改，不用重新开会话。

这些习惯都建立在一个前提上：你在安全的环境里操作。涉及线上和敏感数据，有几件事要提前知道。

## 非技术人做开发，有几件事要注意

工具好用不代表没有风险。非技术人第一次用 Claude Code 做开发，有几个地方容易出问题。

**生产环境不是练手的地方**。Claude Code 能直接操作文件、跑命令、改数据库，它不会主动区分你在沙箱还是在线上。在阿里这类内部复杂环境里，一个写错的 SQL、一条误执行的删除命令、一次对线上配置的修改，都可能造成真实故障。原则是：本地和测试环境随便折腾，涉及生产环境的操作，先在测试环境验证，确认没问题再上。如果 Claude Code 要执行一个你没见过的命令，先停下来搞清楚它在做什么再放行。

**看不懂的操作，先让 Claude Code 解释**。遇到一段命令或一段配置不确定，不要直接让它跑，先问"这段命令具体做了什么，有什么风险"。Claude Code 解释得很清楚，几乎可以作为实时的技术顾问。看懂了再执行，比事后返工省时间。

**复杂操作用 Plan 模式，不要直接执行**。影响多个文件或涉及数据变更的任务，一定先开 Plan 模式看方案，确认改动范围在预期内再执行。不用 Plan 模式直接跑，Claude Code 可能做了你没想到的事，回滚成本很高。

**账号和密钥不要直接粘给 Claude Code**。如果需要配置 API Key、数据库密码这类信息，告诉它放到哪个环境变量或配置文件里，不要直接把明文粘到对话里。Claude Code 的对话历史是存在本地的，但养成习惯更重要。

---

## 最后

这篇讲的所有东西，本质上都在解决同一件事：怎么让 Claude Code 少猜——需求写精确它少猜一次，CLAUDE.md 建好它少猜一次，Skills 把经验固化下来它少猜一次，猜得越少，产出越能直接用，中间不需要你反复兜底和纠偏。

你对业务的判断、对用户的感知、对需求边界的把握，这些是它拿不到的，而恰好也是你作为产品和业务人最不缺的东西，把这些交代清楚，剩下的执行它来跑就好。
